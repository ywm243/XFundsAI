二、整体架构

方案B+

架构选型：独立模块 + 进程内事件总线

在现有智能BI系统基础上，新建独立的询报价模块，共享基础设施（LLM解析、规则引擎、记忆系统），通过进程内事件总线实现Agent间松耦合通信。

选择理由：询报价是有状态多步流程，需要独立状态机管理；银行定价交易有合规隔离要求，独立模块便于审计；共享基础设施避免重复开发；事件总线支持未来扩展市场行情Agent等新能力。

三、询报价核心模块

3.1 报价生命周期状态机

┌──────────────────────────────┐ │ │ ▼ │ [IDLE] ──询价请求──▶ [QUOTING] ──引擎返回──▶ [QUOTED] ──过期──▶ [EXPIRED] │ ┌───────────┼───────────┐ │ │ │ ▼ ▼ ▼ [CANCELLED] [REFRESH] [TRADING] ──同步成交──▶ [TRADED] │ └──失败──▶ [TRADE_FAILED]
状态触发条件可执行动作超时处理
IDLE初始 / 完成后发起询价无
QUOTING用户提交询价请求等待引擎响应5秒 → ERROR
QUOTED引擎返回报价确认下单 / 取消 / 刷新 / 等过期有效期到 → EXPIRED
EXPIRED有效期到重新询价无
TRADING确认下单等待同步成交10秒 → TRADE_FAILED
TRADED成交返回完成无
TRADE_FAILED下单失败重试 / 重新询价无
CANCELLED用户取消重新询价无

3.2 意图分类与执行策略

意图类型示例输入执行策略
SINGLE"美元即期询价"1次询价 → 展示报价
MULTI"美元和欧元即期询价"多次询价并发 → 展示多报价
COMPARE"美元即期和3个月远期比价"多次询价并发 → 对比展示
SCENARIO"美元远期不同期限对比"按预设情景多次询价 → 对比展示
DIRECT_TRADE"帮我买100万美元即期"询价 → 展示报价+确认 → 下单

关键约束：比价和情景分析本质是多次调用真实询价接口，本地对比展示。计价引擎不支持假设参数（如"汇率涨到7.3"），只能用真实交易参数组合询价。

情景由业务人员在规则中配置，不是LLM自由组合：

3.4 风控校验边界

阶段本系统职责代客系统职责
询价前黑名单/冻结客户拦截、客户状态检查—
询价参数组装、调用询价接口计算报价+点差
询价后报价异常检测（价格为0、点差异常大等）—
下单传报价ID、调用交易接口、展示结果授信检查、限额检查、成交/拒绝

下单失败时，代客系统返回拒绝原因（额度不足、授信到期等），本系统翻译成客户可理解的语言并展示。

四、交易流程

4.1 正常流程

[QUOTED] ──用户点击确认──▶ 调代客交易接口(quote_id) ──同步返回──▶ [TRADED] │ 展示成交确认单： · 交易编号 · 成交价格 · 成交时间 · 交易要素摘要

4.2 异常流程

[QUOTED] ──确认──▶ 调交易接口 ──拒绝──▶ [TRADE_FAILED] │ 解析拒绝原因： · CREDIT_EXCEEDED → "可用授信额度不足" · LIMIT_EXCEEDED → "交易限额超限" · CUSTOMER_FROZEN → "客户账户状态异常" · QUOTE_EXPIRED → "报价已过期，请重新询价" · SYSTEM_ERROR → "系统异常，请稍后重试" │ 小白客户 → 额外引导说明 │ 可选操作：重新询价 / 联系客户经理

4.3 直接询价交易

用户没有先询价，直接说"帮我买100万美元即期"，走询价流程但不停留：

用户："帮我买100万美元即期" │ ▼ 解析意图 → 识别为 DIRECT_TRADE 类型 │ ▼ 调询价接口 → 获取报价 │ ▼ 展示报价 + 倒计时 + "确认交易"按钮 │ （不是直接下单，必须经过报价展示环节） ▼ 用户确认 → 调交易接口 → 成交/失败

合规要求：不能跳过报价展示环节。即使客户直接说要交易，也必须先调询价接口拿到报价，展示给客户看（含有效期、风险提示），客户确认后才下单。

4.4 报价有效期处理

  • 有效期在本地规则中配置，默认5分钟，可通过Admin API热更新
  • 前端 QuoteCountdown.vue 实时显示剩余时间
  • 过期前30秒变红提示"报价即将过期"
  • 过期后自动标记EXPIRED，提示重新询价
  • 事件总线发布 quote.expired，市场Agent未来可据此刷新行情

七、LLM解析准确性保障

7.1 五层防线

7.2 必填字段完整性校验

核心原则：缺必填字段 → 追问，不填充默认值，不让LLM猜测。

例外（规则4）：direction 缺失时 → 默认报双边价格（B+S），不追问。客户可点击单边发起下单。

产品类型必填字段可选字段（缺失不追问）
即期(SPOT)currency_pairdirection（缺失→双边报价）
远期(FWD)currency_pair, tenordirection（缺失→双边报价）
掉期(SWAP)currency_pair, near_tenor, far_tenordirection（缺失→双边报价）

7.3 询报价铁律（gatekeep扩展）

铁律规则理由
禁止LLM编造价格所有价格必须来自内部引擎，LLM不能估算、调整、计算盈亏合规红线，不可触碰
产品类型不可推断必须由用户明确说出或上下文继承即期和远期价格差异大
金额不可为0DIRECT_TRADE 意图下金额必须 > 0下单金额为0无意义
期限不可猜测FWD/SWAP的期限必须来自用户输入或上下文1个月和1年远期价差巨大
意图类型保守判定模棱两可时优先归为SINGLE而非DIRECT_TRADE宁可多一步确认

八、代客系统API接口需求

核心依赖

说明：以下接口需要代客系统团队提供，本系统作为调用方。接口设计建议遵循RESTful规范，返回JSON格式数据。

8.1 询价接口

POST /api/v1/pricing/inquiry

功能说明

根据交易要素计算报价，返回价格、点差、报价ID等信息。支持即期、远期、掉期产品。

请求参数

字段名类型必填说明
customer_idstring是客户ID（代客系统客户编号）
product_typestring是产品类型：SPOT(即期) / FWD(远期) / SWAP(掉期)
currency_pairstring是货币对，如 USD/CNY
directionstring是方向：B(银行买入/客户结汇) / S(银行卖出/客户购汇)
amountdecimal否金额（外币金额），不传则返回标准报价
tenorstring远期必填期限，如 1M / 3M / 6M / 1Y
near_tenorstring掉期必填近端期限
far_tenorstring掉期必填远端期限
value_datedate否指定交割日期（远期/掉期），不传则按标准期限推算
request_idstring否请求ID，用于幂等和追踪

返回参数

字段名类型说明

十、合规风控规则

17项规则

10.1 规则总览

#规则影响层级变更幅度
1大额金额分级路由 — 超过阈值不能AI询价/成交，转线下risk_guard + service中
2模拟询价沙盒 — 真实引擎但不下单models + service + trade_executor + 前端中
3情景分析约束 — 不预测/不建议/不承诺，参数来源注明，附带风险提示prompt + 前端低
4默认双边报价 — 客户未指定方向时报双边validator + service + 前端高
5关键数字双表达 — 金额/价格数字+中文大写前端（纯展示）低
6超时会话自动结束 — 5分钟无互动清空记忆state_machine + service中
7拒绝引导系统 — 不支持产品/非交易时段/投资建议必须礼貌拒绝并引导risk_guard + prompt低
8禁止LLM操作价格 — 不能编造、增减点差、计算盈亏prompt低
9客户权限/制裁校验 — 产品权限+黑名单+制裁名单risk_guard + 代客API中
10下单自动计算担保 — 授信占用或保证金处理代客系统 + 前端展示低
11价格不可用处理 — 不展示报价，提示价格异常risk_guard + service低
12客户优惠展示 — 显示本单优惠XX点engine_client + 前端低
13完整审计记录 — 会话/报价/交易全记录含时间戳+引擎原值+LLM决策mysql_store + service中
14电子证据留存 — 报价/风险揭示/交易确认防篡改mysql_store + service中
15情景分析持续提示 — "模拟结果，不代表实际盈亏"前端低
16超限额转人工 — 系统内人工报价，分行确认后返回客户—暂不实现
17频率控制 — 5秒内最多1个价格，前端控制询价间隔risk_guard + 前端中

10.2 大额金额分级路由（规则1）

在 risk_guard.pre_check() 中增加金额阈值判断：

逻辑：金额 > 阈值（如等值5000万美元，可按产品类型配置）→ 返回引导提示，不调询价接口，不下单。

金额范围路由前端展示
≤ 阈值正常AI询价→AI成交正常流程
> 阈值不调询价接口，返回引导"大额交易（等值≥XX万美元）需走线下询价通道，已为您转接客户经理。如需AI快速询价，请减少金额后重试。"

阈值配置在 semantic_rules.json → pricing.amount_thresholds，按产品类型可差异化。

10.3 模拟询价沙盒（规则2）

在 InquiryParams 中增加 sandbox: bool 字段。用户输入含"模拟询价/体验/试算"等关键词时自动启用。

阶段沙盒行为正常行为
询价调真实引擎（体验真实价格）调真实引擎
报价展示蓝色横幅"模拟询价 — 体验流程，不做实际交易"正常展示
下单阻断，提示"模拟模式不支持实际交易"正常下单
会话完成后自动标记CANCELLED正常状态流转
审计标记 sandbox=true正常审计

10.4 情景分析合规约束（规则3 + 规则15）

禁止项：预测价格走势、建议交易时机、承诺收益或亏损。所有情景计算结果必须附带风险提示。

  • 情景计算参数来源必须注明（如"波动率数据来源：XX市场公开数据，仅供参考"）
  • 前端 ScenarioCompare 底部固定红色横幅："模拟结果，不代表实际盈亏。实际成交价格以询价接口返回为准。"
  • PRICING_SYSTEM_PROMPT 中显式禁止预测、建议、承诺

10.5 默认双边报价（规则4）

核心变更：direction 从必填改为可选。缺失时默认并发询价（B+S），展示双边价格。客户可点击单边发起下单。

用户："美元即期询价"（未指定结汇/购汇） │ ▼ validator.validate_intent() → direction缺失，不追问 │ ▼ service._build_inquiry_params() → 生成2组参数： InquiryParams(direction='B') + InquiryParams(direction='S') │ ▼ batch_inquiry([B, S]) → 并发询价 │ ▼ PricingCard 展示双边： ┌──────────────────────────┐ │ 美元/人民币 即期 │ │ 结汇价(B): 7.2350 │ 购汇价(S): 7.2450 │ │ 点差: 100bp │ │ [按结汇价下单] [按购汇价下单] │ └──────────────────────────┘

若客户明确指定方向（如"美元购汇询价"），仍走单边报价流程。

10.6 关键数字双表达（规则5）

纯前端改动，增加 formatChineseAmount() 工具函数，所有金额、价格展示使用双格式：

价格：7.2450 金额：$1,000,000.00（壹佰万美元整） 成交价：7.2450 保证金：¥724,500.00（柒拾贰万肆仟伍佰元整）

10.7 超时会话自动结束（规则6）

  • pricing_sessions 表增加 last_activity 字段
  • 每次请求更新 last_activity
  • 定时检查（或请求时懒检查）：距 last_activity > 5分钟 → 状态转 EXPIRED → 记忆摘要化 → 推送"会话超时，如需继续请重新发起"
  • 配置：semantic_rules.json → pricing.session_timeout_minutes: 5

10.8 拒绝引导系统（规则7）

拒绝类型触发条件引导文案
UNSUPPORTED_PRODUCT期权、结构性产品等"暂不支持[产品名]，当前支持即期、远期、掉期询价"
NON_TRADING_HOURS非交易时段（根据币种）"当前为非交易时段，交易时段为工作日9:00-17:00。您可使用模拟询价体验流程。"
INVESTMENT_ADVICE"会涨吗""什么时候买合适""根据监管要求，我们无法提供投资建议。您可以对比不同产品的价格差异，选择最适合的方案。"
PRICE_PREDICTION"预测""走势""会不会跌""我们无法预测汇率走势。如需了解不同情景下的价格差异，可以使用情景分析功能。"
AMOUNT_TOO_LARGE金额超阈值"大额交易需走线下询价通道。请联系客户经理或减少金额后重试。"

10.9 禁止LLM操作价格（规则8）

在 PRICING_SYSTEM_PROMPT 中增加强制约束：

你绝对不能： 1. 对报价增减点差、调整汇率 2. 自行计算盈亏金额、损益数字 3. 估算或猜测任何价格、汇率 4. 提供任何形式的投资分析或建议 5. 编造不存在的信息来回答客户问题 所有价格数据必须来自内部计价引擎。 当无法获取真实价格时，必须如实告知客户"当前价格异常，暂无法报价"。

10.10 客户权限与制裁校验（规则9）

risk_guard.pre_check() 扩展为三层校验：

层级检查项数据来源失败处理
1制裁名单/黑名单代客系统客户接口（新增 sanctions 字段）静默拒绝，不提示具体原因
2账户状态（冻结/销户）代客系统客户接口拒绝 + 引导联系客户经理
3产品权限（是否开通远期/掉期）代客系统客户接口（新增 product_permissions 字段）拒绝 + 引导开通权限

10.11 下单担保计算（规则10）

代客系统交易接口负责担保计算，本系统只需展示返回结果：

// 交易成功返回中新增 { "trade_id": "FX20260519001", "executed_rate": 7.2450, "margin": { "type": "CREDIT", // CREDIT(授信占用) / CASH(保证金) "amount": 724500.00, "currency": "CNY" } }

前端 trade_success 卡片中展示担保/保证金信息。若 margin.type == "CASH" 且有保证金账户扣款说明，一并展示。

10.12 价格不可用处理（规则11）

场景处理客户提示
引擎返回 PRICE_UNAVAILABLE不展示报价"当前价格异常，暂无法报价，请稍后重试"
引擎超时（>5秒）不展示报价"询价请求超时，请稍后重试"
risk_guard.post_check 异常不展示报价"报价结果校验异常，请联系系统管理员"
引擎返回空结果不展示报价"暂无可用报价，请检查参数后重试"

关键约束：任何情况下都不能对客展示空白/为0/为负的价格，也不能由LLM编造替代价格。

10.13 客户优惠展示（规则12）

引擎返回中增加 discount_bp 字段。本系统不做优惠计算，只展示引擎返回值：

报价：7.2450（已享优惠 20bp） 点差：50bp（原70bp，优惠20bp）

10.14 完整审计与电子证据（规则13 + 规则14）

pricing_audit_log 表扩展：

字段类型说明
engine_raw_responseJSON引擎返回原始值（完整保留，不可篡改）
llm_decision_stepsJSONLLM解析的每一步决策（意图识别→参数提取→结果）
evidence_hashVARCHAR(64)SHA256(pricing_id + action + detail + timestamp)
evidence_typeVARCHAR(32)inquiry / quote / risk_disclosure / trade_confirm

关键操作（报价展示、风险揭示确认、交易确认）生成 evidence_hash，作为防篡改电子证据：

evidence_hash = SHA256( pricing_id + action + json.dumps(detail) + timestamp )

10.15 频率控制（规则17）

后端控制

  • 基于 customer_id 的内存/Redis缓存
  • 记录 last_inquiry_time
  • 5秒内同客户再次询价 → 返回429 "询价过于频繁，请X秒后重试"

前端控制

  • 询价按钮点击后5秒内禁用
  • 刷新按钮同理
  • 倒计时显示剩余等待秒数

配置：semantic_rules.json → pricing.rate_limit_seconds: 5
