# Smart BI 前台页面重设计 — 设计规格

> 版本：v1.0 | 日期：2026-05-14 | 基于 brainstorming 会话确认

---

## 1. 目标

将当前单栏聊天式布局升级为专业、易用的 BI 分析平台界面，保持对话式查询为核心交互，同时支持多 Agent 扩展、查询历史、管理后台。

**核心理念：** 对话式为主 — 用户通过多轮对话完成数据分析，"本月交易量" → "上月呢" → "排名前十"。

## 3. 结果卡片设计

每条 Bot 结果是**一张完整的信息卡片**，四个区域从上到下排列，全部默认展开：

### 3.1 摘要行

- 渐变背景（`#1e3a5f` → `#1e293b`）与卡片主体区分
- NL 一句话总结：金额高亮（`#60a5fa`）、正向变化绿色（`#34d399`）
- 底部元信息行（`#94a3b8` 小字）：
  - 🕐 时间范围
  - 📋 维度/TopN 标签
  - ▲/▼ 同比/环比变化率

### 3.2 图表区

- 小标题 "📈 图表标题"（自动生成，如"交易量趋势对比"）
- 图表类型由 `visualization/chart_detector.py` 自动选择
- 最小高度 140px，ECharts 渲染
- 始终展开，不折叠

### 3.3 数据洞察与建议

- 深色背景（`#1a2230`）与表格区域区分
- 小标题 "💡 数据分析与建议"
- 2-4 条结构化洞察，每条带图标：
  - ⚠️ 黄色 — 风险/集中度提示
  - 📈 绿色 — 增长亮点
  - 📋 蓝色 — 数据质量/建议
- 每条一行，不超过 60 字

### 3.4 数据表格

- Naive UI DataTable，列名中文化，金额格式化
- 新增"同比变化"列（绿色+ / 红色-）
- 底部操作栏：`[数据] [SQL] [参数]` 标签切换 | 共 N 行 | 📥 导出

## 5. 交互行为

### 5.1 查询流程（不变）

```
用户输入 → Enter 发送
  → 用户消息（蓝色气泡，右对齐）
  → 加载态（NPulse + "思考中..."）
  → 确认卡（参数可编辑 + "确认查询"/"重置"）
  → 结果卡片（摘要→图表→洞察→表格）
  → 输入框保持焦点，可继续追问
```

### 5.2 多轮对话（不变）

```
"本月交易量" → 查询结果
"上月呢"     → 继承上下文（product_type/银行/维度），日期前移
"排名前五"   → 继承上下文，叠加 top_n=5
```

### 5.3 侧边栏交互（新增）

- 默认收起（56px），Agent 图标始终可见
- 点击 🕐（历史）：侧边栏展开到 220px，显示最近查询列表
- 点击列表项：恢复该次查询的对话上下文
- 点击对话区空白处：侧边栏收起
- 点击 ⚙（管理）：切换到 AdminRules 视图

### 5.4 首页快捷按钮（新增）

- 点击任意快捷示例 → 等同于输入该文字并按发送
- 进入对话流后，首页引导消失，显示正常对话

### 5.5 Agent 切换（预留，本次不实现）

- 点击 📊 / ⚠️ 无响应（半透明 disabled 态）
- 后续实现后，切换 Agent 会清空当前对话，显示新 Agent 的首页引导

### 7.3 可读性设计（面向不懂代码的业务人员）

**原则：每一条规则必须让业务人员一眼看懂，不需要懂 JSON。**

| 设计要点 | 说明 |
|---------|------|
| 中文标签 | 规则分类名、字段名全部中文展示（"买卖方向"而非"buy_sell_direction"） |
| 关键词可读 | 关键词以逗号分隔的标签形式展示（"结汇, 结汇交易"），并非 JSON 数组 `["结汇","结汇交易"]` |
| 规则数据可读 | 每条规则的值以中文键值对展示，例如 `方向: B（银行买入）`、`设为结售汇: 是`，而非裸 JSON |
| 铁律标识 | 铁律规则显示 ✓ 标记 + 橙色边框，"此规则不可被客户前缀反转、不可被 AI 覆盖" |
| 行内编辑 | 点击"编辑"弹出模态框，每个字段有中文 label + placeholder 示例 |
| 预览测试 | 输入任意中文查询，实时展示规则匹配结果，验证规则是否正确 |
| 帮助提示 | 每个分类顶部有说明文字，例如买卖方向："银行视角：结汇=银行买入(B)，购汇=银行卖出(S)" |
| 版本历史 | 每次修改自动备份，可一键回滚任意历史版本 |

**编辑模态框示例（添加买卖方向规则）：**

### 7.4 后端逻辑校验

**保存前必须通过以下校验，不通过则拒绝并返回中文错误信息：**

| 校验规则 | 说明 | 错误示例 |
|---------|------|---------|
| 关键词唯一性 | 同一分类下关键词不可重复 | "关键词'结汇'已存在于该分类中" |
| 铁律关键词冲突 | 铁律规则的关键词不能与可反转规则重叠 | "关键词'结汇'已作为铁律规则，不能同时作为可反转规则" |
| 方向必填 | buy_sell 类规则 direction 字段不能为空 | "买卖方向规则的'方向'字段不能为空" |
| 状态码范围 | special_states 的 value 必须在 0-5 之间 | "特殊状态值必须在 0-5 范围内" |
| 产品类型合法 | product_type 的 value 只能是 spot/fwd/swap/all | "未知的产品类型: xxx" |
| 优先级整数 | priority 必须是非负整数 | "优先级必须是大于等于0的整数" |
| JSON 格式 | rule_data 必须是合法 JSON（编辑时自动序列化） | "规则数据格式错误，请检查" |
| 必填字段 | 每类规则有各自的必填字段白名单 | "缺少必填字段: direction" |

**校验在前端和后端各执行一次：前端即时提示，后端最终守门。**

### 7.5 热部署流程

```
前端点击 [热部署]
  → 确认弹窗: "确认部署？新规则将立即对所有用户生效"
  → POST /api/admin/rules/reload
     → rules_engine.reload_rules()        → _rules_cache = None
     → prompt_builder.invalidate_cache()  → _cache = None
     → 自动备份当前版本到 rule_versions 表
  → 返回 { status: "ok", version: 42 }
  → 前端显示 "✓ 规则已生效，版本 42"
  → 下一个用户请求自动从 SQLite 加载新规则（延迟 < 5ms）
```

**全流程不需要重启服务，不需要 SSH 登录服务器。**

### 7.6 版本管理

- 每次增/改/删操作自动备份当前分类的所有规则到 `rule_versions` 表
- 版本历史页显示版本号 + 时间，点击"回滚"恢复到指定版本
- 回滚后自动触发热部署（清缓存）
